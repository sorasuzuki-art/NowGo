====================================================
NowGo プラン提案条件（詳細仕様）
====================================================

■ 概要
ユーザーが設定した「遊べる時間」「出発地」「歩ける範囲」「天気」「スタイル」「モード」などの
条件をもとに、周辺のスポットを組み合わせてプラン（2〜6スポット）を自動生成する。

====================================================
1. 入力パラメータ
====================================================

  - availableTime（遊べる時間）: 分単位（例: 120分）
  - currentHour / currentMinute: 開始時刻
  - weather（天気）: 晴れ / 曇り / 雨 / 雪 / 風強め
  - style（スタイル）: ゆっくり / ほどほど / アクティブ
  - locationType: 屋内 / 屋外（未指定なら両方）
  - mode（モード）: 定番 / 新規開拓 / 冒険
  - origin: 出発地の緯度・経度
  - walkRangeMinutes: 歩ける範囲（分）。デフォルト20分
  - excludeSpotIds: 除外するスポットID一覧（既に表示済みのもの）
  - pinnedSpots: ピン留め（ブックマーク）されたスポット（再作成時に引き継ぐ）

====================================================
2. スポット候補の絞り込み（DB検索フェーズ）
====================================================

2-1. 基本条件
  - isActive = true のスポットのみ対象

2-2. 屋内/屋外フィルタ
  - 「屋内」選択時: indoor_type が indoor または both のみ
  - 「屋外」選択時: indoor_type が outdoor または both のみ
  - 未指定: フィルタなし

2-3. バウンディングボックス（エリア絞り込み）
  - 出発地を中心に、walkRangeMinutes 分の距離を半径とする矩形で候補を絞る
  - 距離計算: 徒歩速度 80m/分（= 4.8km/h）
  - 例: walkRangeMinutes=20 → 半径 1.6km の矩形
  - 緯度方向: ±(walkKm / 111)度
  - 経度方向: ±(walkKm / (111 × cos(緯度)))度

2-4. 取得上限
  - 最大500件を取得

2-5. フォールバック
  - バウンディングボックス内に候補が0件の場合、エリア制限を外して最大300件取得

====================================================
3. 個別スポットのスコアリング（scoreSpot）
====================================================

各スポットに対して base score = 1.0 から開始し、以下の条件で加減算する。
スコアが null になるスポットはプラン候補から完全除外される。

3-0. 基本除外
  - 名前(name)、緯度(lat)、経度(lon) のいずれかが欠損 → 除外
  - 営業時間外（開始時刻の時点で閉まっている）→ 除外
  - excludeSpotIds に含まれる → 除外（既に表示済み）
  - 訪問済みスポット（visited_spots テーブル）→ 除外

3-1. 天気による補正
  ┌──────────────────┬─────────────────────────────────┐
  │ 天気              │ 補正内容                         │
  ├──────────────────┼─────────────────────────────────┤
  │ 雨 / 雪          │ 屋外スポット → ×0.1（ほぼ除外）  │
  │                   │ 屋内スポット → ×1.4（優遇）      │
  │                   │ 「雨OK」タグ → ×1.3              │
  │ 晴れ              │ 屋外・両方  → ×1.2               │
  │ 曇り / 風強め     │ 補正なし                         │
  └──────────────────┴─────────────────────────────────┘

3-2. 滞在時間の妥当性チェック
  - 目安スポット数 = availableTime ÷ 35（最低2）
  - 1スポットあたりの適正時間 = availableTime ÷ 目安スポット数
  - スポットの推定滞在時間が適正時間の2倍超 → ×0.5
  - スポットの推定滞在時間が適正時間の1.5倍超 → ×0.8

3-3. スタイルによる補正
  ┌────────────────┬──────────────────────────────────────┐
  │ スタイル        │ 補正内容                              │
  ├────────────────┼──────────────────────────────────────┤
  │ ゆっくり        │ stay型 → ×1.3、short型 → ×0.8       │
  │ アクティブ      │ roam型 → ×1.3、short型 → ×1.2、     │
  │                 │ 屋外 → ×1.1                          │
  │ ほどほど        │ short型 → ×1.1                       │
  └────────────────┴──────────────────────────────────────┘
  ※ stay_type: stay=じっくり滞在型、roam=歩き回り型、short=サクッと型

3-4. モード＆スコープによる補正（最重要）

  ◆ 定番モード
    - スコア = 0.8 + famousLevel × 0.15
    - 除外条件なし（有名スポットほど有利）

  ◆ 新規開拓モード（新たなお店やスポットとの出会いが目的）
    - 【完全除外】famousLevel ≧ 5（超有名スポット）
    - 【完全除外】scope = big（商業ビル・大型エリア・通り）
    - 【完全除外】category = landmark または town
    - famousLevel ≦ 2 → ×1.4（知名度が低いほど優遇）
    - famousLevel = 4  → ×0.4（やや有名 → 大幅ペナルティ）
    - scope = medium → ×0.5（中規模施設はペナルティ）
    - scope = small → 補正なし（個別スポットが理想）

  ◆ 冒険モード
    - 【完全除外】famousLevel ≧ 5
    - 【完全除外】scope = big
    - 【完全除外】category = landmark または town
    - カテゴリが いきもの/エンタメ/リラックス/ミュージアム → ×1.3
    - scope = medium → ×0.5

  ※ scope の定義:
    - big: 商業ビル・大型施設・エリア・通り（例: 東京ミッドタウン、表参道ヒルズ）
    - medium: 中規模の施設（例: 美術館、公園）
    - small: 個別の小さなお店やスポット（例: カフェ、雑貨屋、ギャラリー）

  ※ famousLevel の定義:
    - 1: ほとんど知られていない
    - 2: 一部の人が知っている
    - 3: そこそこ知られている
    - 4: 結構有名
    - 5: 超有名（東京タワー、浅草寺レベル）

3-5. タグによるボーナス
  - ゆっくりスタイル + タグに「のんびり」「リラックス」「静か」 → ×1.15
  - 新規開拓モード + タグに「穴場」「隠れ家」「ローカル」 → ×1.15

3-6. 価格帯による補正
  ┌──────────┬──────────┐
  │ 価格帯    │ 補正      │
  ├──────────┼──────────┤
  │ free      │ ×1.2     │
  │ low       │ ×1.1     │
  │ medium    │ ×1.0     │
  │ high      │ ×0.9     │
  └──────────┴──────────┘

====================================================
4. スポット選択アルゴリズム（selectDiverseSpots）
====================================================

スコアリングを通過した候補から、以下のアルゴリズムでプランに含めるスポットを選ぶ。
ポイント: 「完全ランダム性」と「多様性」を重視。

4-1. 目標スポット数
  - targetSpots = availableTime ÷ 35（切り上げ、最低2、最大6）
  - ピン留めスポット分を差し引いた残りを新規スポットで埋める

4-2. 1つ目のスポット選択
  - 全候補から【完全均等ランダム】で1つ選ぶ
  - スコアによる重み付けなし（バウンディングボックスでエリアは既に絞られている）
  - → 毎回異なるスポットが起点になり、プランのバリエーションが増える

4-3. 2つ目以降のスポット選択（重み付きランダム）
  各候補について以下のスコアを計算し、重み付きランダムで選択する:

    combo = √(スポットスコア) × 距離減衰 × カテゴリ多様性ボーナス

  ◆ √(スポットスコア): スコアの平方根を取ることでスコア差を平坦化
    - 例: スコア1.0→1.0、スコア0.5→0.71、スコア0.1→0.32
    - 高スコアだけに偏らず、ランダム性を確保しつつ最低品質は担保

  ◆ 距離減衰: exp(-距離km × 0.7)
    - 直前のスポットからの距離に応じた緩やかな減衰
    - 1km → 約0.50、1.5km → 約0.35、2km → 約0.25
    - 近いスポットがやや有利だが、遠くても十分候補に入る
    - ハードリミットなし（歩ける範囲外でも確率的に選ばれ得る）

  ◆ カテゴリ多様性ボーナス
    - 既に選ばれたカテゴリと同じ → ×0.05（20倍不利）
    - 新しいカテゴリ → ×1.0
    - → カフェ→ギャラリー→グルメ→自然 のように多様なプランになりやすい

4-4. カテゴリグループ一覧
  ┌──────────────┬────────────────┐
  │ DBカテゴリ    │ 表示グループ    │
  ├──────────────┼────────────────┤
  │ cafe          │ カフェ          │
  │ bar           │ バー            │
  │ clothes       │ ファッション    │
  │ zakka         │ 雑貨            │
  │ shopping      │ ショッピング    │
  │ museum        │ ミュージアム    │
  │ gallery       │ ギャラリー      │
  │ garden, park  │ 自然            │
  │ amusement,    │ エンタメ        │
  │  entertainment│                 │
  │ zoo, aquarium │ いきもの        │
  │ temple        │ 寺社            │
  │ town          │ まち歩き        │
  │ spa           │ リラックス      │
  │ restaurant    │ グルメ          │
  │ landmark      │ ランドマーク    │
  │ sports        │ スポーツ        │
  └──────────────┴────────────────┘

====================================================
5. ルート最適化（optimizeRoute）
====================================================

選ばれたスポットを最短距離で回れるように並び替える。

5-1. 同一ビルグルーピング
  - 住所からビル名を自動パース
  - 同じビル内のスポットは1つの「ノード」として扱い、必ず連続配置
  - ビル内の移動時間は0分として計算

5-2. 全順列探索
  - ノード数が少ない（通常6以下）ため、全ての訪問順序を試して最短距離を選択
  - 出発地 → 1番目 → 2番目 → ... の合計距離が最小になる順序を採用

====================================================
6. タイムテーブル生成（buildPlan）
====================================================

6-1. 開始時刻の丸め
  - 開始分を15分単位に切り上げ（例: 14:22 → 14:30）

6-2. 移動時間の計算
  - 出発地 → 最初のスポット: 直線距離 ÷ 80m/分（最低3分）
  - スポット間: 直線距離 ÷ 80m/分（最低3分）
  - 同一ビル内: 0分

6-3. 滞在時間の決定
  - 各スポットのDBの estimated_stay_min をそのまま使用（キャップなし）
  - DBに値がない場合のデフォルト: 30分

6-4. 時間制限（デッドライン方式）
  - デッドライン = 開始時刻 + availableTime × 1.1（10%のオーバーは許容）
  - 各スポットを追加する前に「移動+滞在がデッドラインを超えないか」チェック
  - 超える場合は残りのスポットを打ち切り
  - ただし、2スポット未満の場合は超えても追加する
  - ピン留めスポットは時間制限に関係なく必ず含める

====================================================
7. 除外リスト管理（再作成時）
====================================================

7-1. 基本動作
  - プランを「再作成」するたびに、前回表示したスポットIDを除外リストに追加
  - → 再作成のたびに新しいスポットが表示される

7-2. 候補枯渇時のリセット
  - 除外リストが溜まりすぎて候補が0件になった場合:
    1. 除外リストを「現在のプランのスポットのみ」にリセット
    2. リセットした除外リストで再検索
    → 以前表示したスポットも再び候補に戻る

7-3. 訪問済みスポットの除外
  - ログインユーザーの場合、visited_spots テーブルに記録されたスポットも除外
  - プラン完了時に全スポットが visited_spots に記録される

====================================================
8. ピン留め（ブックマーク）の扱い
====================================================

  - ユーザーがプラン画面でピン留めしたスポットは、再作成時も必ず含まれる
  - ピン留めスポットはスコアリングを経ず、最高スコア（10）で固定
  - 滞在時間のキャップも適用されない（元の推定時間を維持）
  - 時間制限（デッドライン）に関係なく必ずプランに含まれる
  - ルート最適化には参加する（最適な位置に配置される）
