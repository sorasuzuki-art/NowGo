# 仮説検証システム設計書

> **目的**: NowGo-仮説検証.md に記載された25個の仮説を、システムの仕組みで検証可能にする

---

## 仮説の分類

25個の仮説を**検証手段**で2つに分類する。

| 分類 | 検証手段 | 対象仮説 |
|------|----------|----------|
| **A. ログ分析型** | アプリ内の行動データで検証 | H6, H7, H10, H13, H14, H17, H18, H20, H21, H22, H23, H24, H25 |
| **B. インタビュー型** | ユーザーの主観・意見が必要 | H1, H2, H3, H4, H5, H8, H9, H11, H12, H15, H16, H19 |

- **A（ログ分析型）** はシステムだけで完結できる
- **B（インタビュー型）** もアプリ内アンケート機能で一部システム化できる

## 構築する3つのコンポーネント

### 1. イベントトラッキング基盤

現状、アプリには行動ログの仕組みが一切ない。
Supabase に `analytics_events` テーブルを作り、ユーザーの行動を記録する。

#### テーブル設計

```sql
-- analytics_events
id            UUID PRIMARY KEY DEFAULT gen_random_uuid()
user_id       UUID REFERENCES auth.users(id) -- nullable（ゲストは anonymous ID）
session_id    TEXT NOT NULL                   -- セッション識別
event_name    TEXT NOT NULL                   -- イベント名
event_params  JSONB DEFAULT '{}'              -- 柔軟なパラメータ
screen        TEXT                            -- どの画面で発生したか
created_at    TIMESTAMPTZ DEFAULT now()
```

#### 記録するイベント一覧

| イベント名 | 内容 | パラメータ例 | 検証対象 |
|-----------|------|-------------|---------|
| `app_open` | アプリ起動 | `{ hour, dayOfWeek }` | H6 (利用時間帯) |
| `plan_generation_start` | プラン生成開始 | `{ source: 'dashboard' \| 'quickplan' }` | H25 (完了率) |
| `plan_generation_complete` | プラン生成完了 | `{ mode, duration, weather, style, walkRange, spotCount }` | H13, H21, H22 |
| `plan_param_set` | パラメータ設定 | `{ param, value }` | H20, H21, H22, H14 |
| `plan_regenerate` | プラン再作成 | `{ attemptNumber, pinnedCount }` | H24 (再作成使用率) |
| `plan_execution_start` | プラン実行開始 | `{ spotCount, totalDuration }` | H25 |
| `plan_execution_complete` | プラン完了 | `{ spotsVisited, totalSpots, elapsedMin }` | H25 |
| `plan_execution_abort` | プラン途中終了 | `{ spotsVisited, totalSpots, abortAtSpotIndex }` | H25 |
| `spot_visited` | スポット訪問記録 | `{ spotId, category, stayMin }` | H7, H10 |
| `spot_excluded` | 除外リスト使用 | `{ spotId, reason }` | H10, H23 |
| `exclusion_reset` | 除外リストリセット | `{ previousCount }` | H23 |
| `guest_signup` | ゲスト→会員登録 | `{ plansCreatedAsGuest }` | H17 (転換率) |
| `screen_view` | 画面遷移 | `{ screen, previousScreen }` | ファネル分析全般 |
| `favorite_spot` | スポットお気に入り | `{ spotId, category }` | H18 |
| `spot_category_in_plan` | プラン内のスポットカテゴリ | `{ categories: string[] }` | H7 (飲食店含有率) |
| `mode_selected` | モード選択 | `{ mode, isDefault }` | H13, H20 |
| `duration_selected` | 時間選択 | `{ minutes }` | H21 |
| `weather_condition_set` | 天気条件設定 | `{ weather, isChanged }` | H22 |
| `walk_range_selected` | 徒歩範囲選択 | `{ minutes }` | H14 |

#### 実装方針

```
lib/analytics.ts に集約

trackEvent('plan_generation_complete', { mode: '定番', duration: 120 })

のように各コンポーネントから1行で呼べる設計にする。
```

- ゲストユーザーは `crypto.randomUUID()` で匿名IDを生成し localStorage に保存
- 会員登録時に匿名IDと user_id を紐付け（H17 の検証に必須）
- イベント送信は非同期でUIをブロックしない
- 送信失敗時はローカルにバッファし、次回起動時にリトライ

### 2. アプリ内アンケート機能

特定タイミングでユーザーに1〜2問の短いアンケートを表示する仕組み。
インタビュー型仮説をスケーラブルに検証する。

#### テーブル設計

```sql
-- surveys（アンケート定義）
id                UUID PRIMARY KEY DEFAULT gen_random_uuid()
hypothesis_id     TEXT NOT NULL           -- 対応する仮説番号（例: 'H1', 'H8'）
question          TEXT NOT NULL           -- 質問文
options           JSONB NOT NULL          -- 選択肢の配列
trigger_condition TEXT NOT NULL           -- 表示条件
max_responses     INT                     -- 最大回答数（nullで無制限）
is_active         BOOLEAN DEFAULT true
created_at        TIMESTAMPTZ DEFAULT now()

-- survey_responses（回答）
id          UUID PRIMARY KEY DEFAULT gen_random_uuid()
survey_id   UUID REFERENCES surveys(id)
user_id     UUID                         -- nullable（ゲスト対応）
answer      JSONB NOT NULL               -- 回答内容
created_at  TIMESTAMPTZ DEFAULT now()
```

#### 表示タイミングと質問の対応

| 表示タイミング | 質問例 | 検証対象 |
|---------------|--------|---------|
| 初回プラン生成後 | 「遊ぶ前に一番面倒なことは？」（選択肢: 場所探し / プラン組み / 移動手段調べ / その他） | H1, H9 |
| 初回プラン生成後 | 「プランに飲食店が入っていてほしいですか？」 | H7, H11 |
| プラン完了後 | 「スポット1つの提案と、流れのあるプラン、どちらが嬉しい？」 | H8 |
| プラン途中終了時 | 「途中でやめた理由を教えてください」（選択肢: 時間がなくなった / 行きたい場所がなかった / 疲れた / その他） | H25 |
| 3回目利用時 | 「"どこ行く？"を考える時間はどう感じますか？」 | H2 |
| プラン実行中 | 「予算感がわかるといいですか？」 | H15 |
| 5回目利用時 | 「一人 / 友達 / デートで行き先は変わりますか？」 | H12 |

#### 実装方針

- 1回のセッションで表示するアンケートは最大1つ
- 同じアンケートに2回回答させない（回答済みフラグ管理）
- 画面下部にスライドインするシンプルなUIで、閉じるボタンあり
- 「スキップ」も行動データとして記録する

### 3. 仮説検証ダッシュボード

収集したデータを仮説ごとに可視化する管理画面。

#### 構成

- `/admin/analytics` として管理画面に配置（一般ユーザーには非公開）
- 仮説ごとにカードを表示し、対応する指標をグラフ化
- Recharts（既にインストール済み）を活用

#### 表示する指標

| 仮説 | 指標 | 可視化方法 |
|------|------|-----------|
| H6 | 時間帯別利用数 | ヒストグラム（0〜23時） |
| H13 | モード選択比率 | 円グラフ（定番 / 新規開拓 / 冒険） |
| H14 | 徒歩範囲の設定値分布 | ヒストグラム |
| H17 | ゲスト→会員の転換率 | ファネルチャート |
| H20 | ユーザーごとのモード選択パターン | ばらつきグラフ |
| H21 | 時間設定値の分布 | ヒストグラム |
| H22 | 天気・屋内外条件の設定率 | 使用率バー |
| H24 | 再作成ボタン使用率 | 割合表示 |
| H25 | プラン完了率・途中終了率 | ファネルチャート |
| H7 | プラン内の飲食店カテゴリ出現率 | 割合バー |
| H18 | 訪問履歴数 × 継続利用率 | 散布図 |

## 仮説ごとの検証方法まとめ

### ログ分析型（A）

| 仮説 | 検証内容 | 必要なイベント | 判定基準 |
|------|---------|---------------|---------|
| H6 | 利用時間帯の分析 | `app_open`, `plan_generation_complete` | 15〜18時が全体の50%以上 → 正しい |
| H7 | 飲食店の選択率 | `spot_category_in_plan` | 飲食カテゴリが10%未満 → 除外でOK |
| H10 | 除外リスト使用率 | `spot_excluded` | 50%以上のユーザーが使用 → 正しい |
| H13 | モード選択比率 | `mode_selected` | 3モードが均等に使われている → 正しい |
| H14 | 徒歩範囲の設定値 | `walk_range_selected` | 80%以上が30分以内 → 徒歩圏内で十分 |
| H17 | ゲスト→会員転換率 | `guest_signup` | プラン実行後の転換率が20%以上 → 正しい |
| H18 | 訪問履歴数 × 継続率 | `spot_visited` + リテンション計算 | 正の相関 → 正しい |
| H20 | モード固定化 | `mode_selected`（ユーザー別集計） | 70%以上が同じモード → 正しい |
| H21 | 時間設定の分布 | `duration_selected` | 120〜180分が70%以上 → ざっくり選択 |
| H22 | 天気条件の使用率 | `weather_condition_set` | 変更率が20%未満 → 使われていない |
| H23 | 除外リストリセット率 | `exclusion_reset` | 10回以上利用者の50%がリセット → 枯渇感あり |
| H24 | 再作成ボタン使用率 | `plan_regenerate` | 使用率20%未満 → 不要の可能性 |
| H25 | プラン完了率 | `plan_execution_complete`, `plan_execution_abort` | 完了率50%未満 → 途中終了が多い |

### インタビュー型（B）→ アプリ内アンケートで補完

| 仮説 | アンケート質問 | 判定基準 |
|------|--------------|---------|
| H1 | 遊ぶ前に一番面倒なことは？ | 「プラン組み」が最多 → 正しい |
| H2 | "どこ行く？"を考える時間はどう感じる？ | 「無駄」が60%以上 → 正しい |
| H3 | 1スポット決めた後どうしてる？ | 「また探す」が多い → 正しい |
| H4 | SNSで保存したスポットに実際行く？ | 「行かない」が60%以上 → 正しい |
| H5 | 外出を諦めた経験は？ | 「ある」が70%以上 → 正しい |
| H8 | スポット1つ vs 流れのあるプラン | 「プラン」が70%以上 → 正しい |
| H9 | 遊ぶ前のどのステップが面倒？ | 「エリア内プラン」が最多 → 正しい |
| H11 | バー・軽食がプランに入っていたら？ | 「嬉しい」が60%以上 → 正しい |
| H12 | 一人/友達/デートで内容変わる？ | 「変わらない」が60%以上 → 正しい |
| H15 | プランを見た時に予算が気になる？ | 「気になる」が60%以上 → 正しい |
| H16 | 全部おまかせで決めてほしい？ | 「はい」が60%以上 → 正しい |
| H19 | （B2Bヒアリングのため対象外） | — |

## 実装の優先順位

```
Phase 1（最優先）: イベントトラッキング基盤
├── Supabase テーブル作成（analytics_events）
├── lib/analytics.ts ユーティリティ作成
├── 匿名ID生成・管理
├── 各画面へのイベント埋め込み
└── 所要: 全画面にイベント送信コードを追加

Phase 2: ログ分析型仮説の計測
├── 集計用 SQL クエリの作成
├── 管理画面（/admin/analytics）の構築
├── Recharts でグラフ可視化
└── 即座に検証可能: H6, H13, H17, H20, H21, H22, H24, H25

Phase 3: アプリ内アンケート機能
├── Supabase テーブル作成（surveys, survey_responses）
├── アンケート表示コンポーネント
├── トリガー条件の判定ロジック
└── 検証可能: H1, H2, H3, H4, H5, H8, H9, H11, H12, H15, H16
```

## 技術的な方針

| 方針 | 理由 |
|------|------|
| **外部サービス（Mixpanel等）は使わず Supabase で完結** | 既存スタックに乗せてコスト・複雑性を抑える |
| **イベント送信は `lib/analytics.ts` に集約** | 各コンポーネントから1行で呼べる設計 |
| **ゲストユーザーも匿名IDで追跡** | H17（転換率）の検証に必須 |
| **RLS（Row Level Security）設定** | analytics テーブルは insert のみ許可、読み取りは管理者のみ |
| **イベント送信は非同期・ノンブロッキング** | ユーザー体験に影響を与えない |
| **送信失敗時はローカルバッファ → リトライ** | データの欠損を最小化 |
