# NowGo 基本設計書

## 1. システム構成

### 1.1 アーキテクチャ概要

```
┌─────────────────────────────────────────────────┐
│                  クライアント                      │
│   Next.js 13 (App Router) + React 18 + Zustand   │
│   ┌───────────┐ ┌──────────┐ ┌────────────────┐  │
│   │ Components│ │  Hooks   │ │      Lib       │  │
│   │ (画面)     │ │ (状態)   │ │ (ビジネスロジック)│  │
│   └─────┬─────┘ └────┬─────┘ └───────┬────────┘  │
│         │            │               │            │
│         └────────────┴───────┬───────┘            │
│                              │                    │
└──────────────────────────────┼────────────────────┘
                               │ HTTPS
                ┌──────────────┴──────────────┐
                │      Supabase (BaaS)        │
                │  ┌──────────┐ ┌──────────┐  │
                │  │PostgreSQL│ │   Auth   │  │
                │  │ (DB)     │ │  (認証)   │  │
                │  └──────────┘ └──────────┘  │
                └─────────────────────────────┘
```

- **サーバーサイドAPIなし**: 全ロジックをクライアントサイドで実行
- **BaaS**: Supabaseが認証・DB・RLSを提供
- **ホスティング**: Netlifyで静的サイトとしてデプロイ

### 2.1 認証機能 (F-AUTH)

#### 概要
Supabase Authを利用したメール/パスワード認証 + ゲストモード

#### コンポーネント構成
- `AuthScreen.tsx` — 認証画面UI
- `lib/auth-context.tsx` — AuthProvider（認証状態管理）

#### データフロー

```
サインアップ:
  AuthScreen → signUp(email, password, nickname, {gender, birthDate})
    → Supabase Auth: createUser
    → DBトリガー: handle_new_user → user_profiles INSERT
    → AuthContext: user + profile をセット
    → setScreen('dashboard')

ログイン:
  AuthScreen → signIn(email, password)
    → Supabase Auth: signInWithPassword
    → AuthContext: user_profiles から profile を取得
    → setScreen('dashboard')

ゲスト:
  AuthScreen → signInAsGuest()
    → AuthContext: isGuest = true（DB操作なし）
    → setScreen('dashboard')
```

#### 画面レイアウト
- 背景画像 + 半透明オーバーレイ
- ゲストモード中のみ「ゲストに戻る」ボタンを表示（← dashboard）
- ログイン/サインアップ タブ切り替え
- フォーム: メール、パスワード（+ サインアップ時: ニックネーム、性別、生年月日）
- 「ゲストで続ける」ボタン

### 2.2 プラン生成機能 (F-PLAN)

#### 概要
ユーザーの検索条件に基づき、Supabase DBからスポットを検索し、スコアリング・ルート最適化を経てプランを自動生成する。

#### コンポーネント構成
- `DashboardScreen.tsx` — 検索条件入力UI
- `lib/spotSearch.ts` — 検索・スコアリング・ルート最適化ロジック

#### 入力項目

| 項目 | 必須 | UI | デフォルト |
|------|------|-----|-----------|
| 出発地 | YES | GPS取得 or 駅名検索 | - |
| 遊べる時間 | YES | ボタン選択 | 120分 |
| モード | NO | ボタン選択 | 新規開拓 |
| 歩ける範囲 | NO | ボタン選択 | 20分 |
| 日付 | NO | 7日間グリッド | 今日 |
| 開始時刻 | NO | ドラムピッカー | 現在時刻 |
| スタイル | NO | ボタン選択 | - |
| 天気 | NO | アイコンボタン | 晴れ |
| 屋内/屋外 | NO | ボタン選択 | 両方 |

#### 処理フロー

```
1. handleGo() ← NowGo!ボタン押下
2. searchSpotsFromDB(params) 呼出
   2-1. visited_spots 取得（ログイン時）
   2-2. Supabase検索（バウンディングボックス、indoor_type、isActive）
   2-3. 各スポットにスコアリング（scoreSpot）
   2-4. 多様性重視の選択（selectDiverseSpots）
   2-5. ルート最適化（optimizeRoute）
   2-6. タイムテーブル生成（buildPlan）
3. setPlan(plan) → Zustandストアに保存
4. setScreen('plan') → プラン確認画面へ
```

#### 駅名検索（オートコンプリート）
- Supabase `stations` テーブルを検索（153駅: JR・東京メトロ・都営・私鉄各線）
- DashboardScreen: name, name_kana でOR部分一致、最大8件、0.2秒デバウンス
- StationSelectModal: 東京都・神奈川県・埼玉県・千葉県の全駅をロード、クライアント側フィルタ

### 2.3 プラン表示機能 (F-VIEW)

#### 概要
生成されたプランをカード+地図で可視化し、ピン留め・再作成を可能にする。

#### コンポーネント構成
- `PlanScreen.tsx` — プラン表示・操作UI
- `PlanMap.tsx` — Leaflet地図

#### 画面構成

```
┌─────────────────────────────┐
│ ヒーローセクション            │
│ (開始時刻 / 所要時間 / N箇所) │
├─────────────────────────────┤
│ 地図（番号マーカー + ルート） │
├─────────────────────────────┤
│ スポットカード #01           │
│ [名前] [カテゴリ] [時刻]     │
│ [説明] [リンク] [ピン留め]   │
├── 移動テキスト ─────────────┤
│ スポットカード #02           │
│ ...                         │
├─────────────────────────────┤
│ [このプランで行く] [再作成]  │
└─────────────────────────────┘
```

#### 戻るボタン
- 確認ダイアログを表示:「ホームに戻りますか？現在のプランは失われます。」
- 「キャンセル」→ そのまま留まる
- 「戻る」→ ダッシュボードへ遷移

#### 再作成ロジック
1. 現在のピン留めスポットを保持
2. これまで表示した全スポットIDを `excludeSpotIds` に追加
3. `searchSpotsFromDB` で新しいスポットを取得
4. 候補0件の場合 → 除外リストを現在のプランのみにリセットして再検索
5. 画面トップへスクロール

### 2.4 プラン実行機能 (F-EXEC)

#### 概要
プランをステップバイステップで実行し、各スポットへのナビゲーション・訪問記録を管理する。

#### コンポーネント構成
- `ExecutionScreen.tsx` — 実行UI

#### 画面構成

```
┌─────────────────────────────┐
│ ヘッダー (戻る / Now / 2/5)  │
│ プログレスバー               │
├─────────────────────────────┤
│ 現在のスポット（大きく表示）  │
│ [スポットへ行く → Maps]      │
├── 移動テキスト ─────────────┤
│ 次のスポット（プレビュー）    │
├─────────────────────────────┤
│ 残りのスケジュール（折畳み）  │
├─────────────────────────────┤
│ [＜前へ] [終了する] [次のスポットへ]│
└─────────────────────────────┘
```

#### ボトムバー操作
- **前のスポットへ（＜）**: `currentSpotIndex > 0` の場合のみ表示。前のスポットに戻る（DB操作なし）
- **終了する**: `currentSpotIndex === 0` の場合のみテキスト表示。途中終了確認ダイアログを表示
- **次のスポットへ**: 次のスポットへ進む（DB操作なし）
- **プランを完了する**: 最後のスポットで表示。完了確認ダイアログを表示

#### Google Mapsナビゲーション
- 開始場所: 常に現在地（origin パラメータなし → デバイスGPS使用）
- 目的地: スポットの緯度・経度
- 移動手段: 徒歩（travelmode=walking）
- URL形式: `https://www.google.com/maps/dir/?api=1&destination={lat},{lng}&travelmode=walking`

#### 完了処理（exitMode による分岐）

**プラン完了（exitMode = 'complete'）**:
1. `markSpotsVisited(全スポットのID)` → visited_spots テーブルに全スポット記録
2. `savePlanHistory(plan)` → plan_history テーブルに記録
3. `setScreen('dashboard')` → ダッシュボードへ戻る

**途中終了（exitMode = 'abort'）**:
1. `markSpotsVisited(currentSpotIndex+1件分のID)` → 現在地までのスポットのみ訪問記録
2. `setScreen('dashboard')` → ダッシュボードへ戻る（プラン履歴は保存しない）

### 2.6 データ永続化機能 (F-DATA)

#### 概要
ログインユーザーはSupabase DB、ゲストはlocalStorageにデータを保存

#### 実装箇所
- `lib/storage.ts` — 全永続化ロジック

#### 保存先の使い分け

| データ | ログイン時 | ゲスト時 |
|--------|-----------|---------|
| 直近の検索駅 | user_settings.recent_stations | localStorage |
| 前回の検索条件 | user_settings | localStorage |
| お気に入り | favorite_spots テーブル | 保存不可 |
| 訪問履歴 | visited_spots テーブル | 保存不可 |
| プラン履歴 | plan_history テーブル | 保存不可 |

## 3. 状態管理設計

### 3.1 Zustand ストア (useNowgoStore)

```typescript
interface NowgoState {
  // 画面制御
  currentScreen: ScreenType;

  // 検索条件
  startLocation: {
    label: string;
    lat: number | null;
    lng: number | null;
    source: 'gps' | 'manual';
    accuracy: number | null;
  };
  duration: number;           // デフォルト: 120
  groupSize: number;          // デフォルト: 2
  walkRangeMinutes: number;   // デフォルト: 20

  // 生成されたプラン
  currentPlan: {
    spots: PlanSpot[];
    startTime: string;
    totalDuration: number;
    pinnedSpots: string[];
    searchParams: SearchParams;
  } | null;
}
```

### 3.2 認証コンテキスト (AuthContext)

```typescript
interface AuthContextType {
  user: User | null;
  profile: UserProfile | null;
  loading: boolean;
  isGuest: boolean;
  signIn: (email, password) => Promise<void>;
  signUp: (email, password, nickname, meta) => Promise<void>;
  signOut: () => Promise<void>;
  signInAsGuest: () => void;
  updateProfile: (updates) => Promise<void>;
}
```
