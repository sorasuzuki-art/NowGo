## 1. テスト概要

### 1.1 目的

本ドキュメントは、NowGo Web Application におけるシステムテスト(ST: System Test)の仕様を定義する。実際のユーザー操作に基づくエンドツーエンドシナリオを通じて、システム全体が要件を満たし、ユーザーが期待通りにサービスを利用できることを検証する。

### 1.2 テスト範囲

| カテゴリ | 対象 |
|---------|------|
| ユーザーシナリオ (ST-SCN) | 新規ユーザー、リピーター、ゲスト、天気/時間/時間長による振る舞い |
| 画面遷移シナリオ (ST-NAV) | 全画面の正常遷移パス、戻るボタン、リロード、不正状態復帰 |
| 異常系シナリオ (ST-ERR) | ネットワークエラー、GPS失敗、DB接続エラー、0件、セッションタイムアウト、不正入力 |
| 性能テスト (ST-PERF) | レスポンスタイム、レンダリング時間 |
| セキュリティテスト (ST-SEC) | RLSポリシー、ゲスト操作制限、トークン期限切れ |

### 1.3 テスト対象システム構成

| レイヤー | 技術要素 |
|----------|----------|
| フロントエンド | Next.js (App Router), React 18, TypeScript, Tailwind CSS |
| 状態管理 | Zustand |
| 認証 | Supabase Auth (Email/Password, Guest Mode) |
| データベース | Supabase (PostgreSQL) + Row Level Security |
| 地図 | Leaflet (PlanMap コンポーネント, dynamic import) |
| ジオコーディング | OpenStreetMap Nominatim API |
| ホスティング | Netlify |

### 1.4 テストID命名規則

```
ST-{シナリオ略称}-{連番3桁}
```

| シナリオ略称 | 対象領域 |
|-------------|---------|
| SCN | ユーザーシナリオ |
| NAV | 画面遷移シナリオ |
| ERR | 異常系シナリオ |
| PERF | 性能テスト |
| SEC | セキュリティテスト |

### 1.5 優先度定義

| 優先度 | 定義 | テスト実施タイミング |
|--------|------|-------------------|
| P1 (Critical) | サービス提供に必須。不合格の場合リリース不可 | 全リリース前に必須実施 |
| P2 (High) | ユーザー体験に大きく影響。代替手段なし | リリース前および重要変更時に実施 |
| P3 (Medium) | ユーザー体験に影響するが代替手段あり | 定期テストサイクルで実施 |
| P4 (Low) | エッジケース。発生頻度が低い | 大規模リリース時に実施 |

### 1.6 合否判定基準

| 判定 | 基準 |
|------|------|
| Pass | 全期待結果を満たし、操作手順通りに完了する |
| Conditional Pass | 軽微な表示崩れはあるが機能的に問題なし (P4以下の不具合) |
| Fail | 期待結果の1つ以上を満たさない。または操作が完了しない |
| Block | 前提条件が満たせずテスト実施不可 |
| Skip | テスト対象外と判断 (根拠を記録) |

### ST-SCN-004: 雨の日のプラン生成 (屋内スポット優先)

| 項目 | 内容 |
|------|------|
| テストID | ST-SCN-004 |
| テスト名 | 雨天時のプラン生成と屋内スポット優先検証 |
| テスト観点 | 天気「雨」を選択した場合、スコアリングにより屋内スポットが優先され、屋外スポットが低スコアになること |
| 優先度 | P2 (High) |

**前提条件:**
1. ログイン済みでDashboardScreen表示中
2. `spot`テーブルに`indoor_type`が`indoor`, `outdoor`, `both`のスポットが混在していること

**操作手順:**

| ステップ | 操作 | 入力値/操作対象 | 確認ポイント |
|---------|------|---------------|-------------|
| 1 | 出発地を設定 | 渋谷駅 | - |
| 2 | 遊ぶ時間を「120分」に設定 | - | - |
| 3 | 「もっと絞り込む」を展開 | - | - |
| 4 | 天気を「雨」に変更 | 「雨」ボタン (CloudRainアイコン) | 選択状態 |
| 5 | 「Go」ボタンをクリック | - | プラン生成 |
| 6 | PlanScreenのスポットカテゴリを確認 | - | 屋内系カテゴリ |
| 7 | 同条件で天気「晴れ」に変更して再検索 | - | 比較用 |

**期待結果:**
1. 天気='雨'のスコアリングが適用されること:
   - `indoor_type === 'outdoor'`: `score *= 0.1` (大幅減点)
   - `indoor_type === 'indoor'`: `score *= 1.4` (ブースト)
   - `weather_ok === '雨OK'`: `score *= 1.3` (ブースト)
2. 生成されたプランのスポットの過半数が屋内系カテゴリ (カフェ、ミュージアム、ショッピング等) であること
3. 純粋な屋外スポット (公園等) がプランに含まれる確率が低いこと
4. 天気='晴れ'のプランと比較して、屋内比率が明確に高いこと

**合否判定基準:**
- Pass: 雨天プランの屋内スポット比率が70%以上
- Conditional Pass: 屋内スポット比率が50-70% (ランダム要素による)
- Fail: 屋外スポットが過半数を占める

| 7 | ニックネーム編集で空文字保存 | nickname: '' | 保存がスキップされる |
| 8 | ニックネーム編集で20文字超 | nickname: 'a'.repeat(21) | maxLength=20で入力制限 |

**期待結果:**
1. ニックネーム空欄: `if (!nickname.trim())`でエラーがthrowされ、「ニックネームを入力してください」が表示されること
2. 不正メール: HTML5の`type="email"`バリデーションでフォーム送信がブロックされること
3. 短いパスワード: HTML5の`minLength={6}`バリデーションでフォーム送信がブロックされること
4. 未登録メール: Supabase Authのエラーレスポンスがcatchされ、エラーメッセージが赤枠内に表示されること
5. 誤パスワード: 同上
6. 特殊文字: SupabaseクエリのパラメータバインディングによりSQL Injection/XSSが防止されること。Reactの自動エスケープによりHTML注入が防止されること
7. 空文字ニックネーム: `if (!nicknameInput.trim()) return;`で早期リターンすること
8. 長すぎるニックネーム: `maxLength={20}`でブラウザ側で入力制限されること

**合否判定基準:**
- Pass: 全パターンでバリデーションが正しく動作し、不正データがDBに保存されない
- Fail: バリデーションなしで不正データが処理される、またはXSS脆弱性

## 6. ST-SEC: セキュリティテスト

### ST-SEC-001: RLSポリシーによる他ユーザーデータアクセス不可

| 項目 | 内容 |
|------|------|
| テストID | ST-SEC-001 |
| テスト名 | Row Level Security (RLS) ポリシーによるデータ分離検証 |
| テスト観点 | SupabaseのRLSポリシーにより、ログインユーザーが自分以外のユーザーのデータ (プロフィール/お気に入り/訪問履歴/プラン履歴) にアクセスできないこと |
| 優先度 | P1 (Critical) |

**前提条件:**
1. テストユーザーA (UserA) がログイン済み
2. テストユーザーB (UserB) のデータが以下のテーブルに存在する:
   - `user_profiles`
   - `favorite_spots`
   - `visited_spots`
   - `plan_history`
   - `user_settings`

**操作手順:**

| ステップ | 操作 | 確認ポイント |
|---------|------|-------------|
| 1 | UserAでログイン | - |
| 2 | DevToolsのコンソールでSupabase SDKを使用してUserBのプロフィール取得を試行 | `supabase.from('user_profiles').select('*').eq('id', 'UserB_ID')` |
| 3 | UserBのお気に入り取得を試行 | `supabase.from('favorite_spots').select('*').eq('user_id', 'UserB_ID')` |
| 4 | UserBの訪問履歴取得を試行 | `supabase.from('visited_spots').select('*').eq('user_id', 'UserB_ID')` |
| 5 | UserBのプラン履歴取得を試行 | `supabase.from('plan_history').select('*').eq('user_id', 'UserB_ID')` |
| 6 | UserBの設定取得を試行 | `supabase.from('user_settings').select('*').eq('user_id', 'UserB_ID')` |

**期待結果:**
1. 全クエリでRLSポリシーにより`data`が空配列 (`[]`) または`null`で返されること
2. エラーレスポンスではなく、空の結果セットが返されること (RLSの標準動作)
3. UserBのnicknameやemailなどの個人情報が取得できないこと
4. `spot`テーブル (公開データ) は正常にアクセスできること (RLSが適用されない、またはSELECTが全ユーザーに許可)

**合否判定基準:**
- Pass: 全テーブルでUserBのデータが取得不可
- Fail: 1テーブルでもUserBのデータが取得可能 (重大セキュリティ問題)
